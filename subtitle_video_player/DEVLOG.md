## 2026-06-16

### 1. 发布渠道：GitHub Release（已落地，commit `aaa1c18`）

- **背景**：构建产物（单文件 HTML）需要分发出去。原本只通过 GitHub Action 推送到 DUFS。
- **决策**：在 workflow 里、DUFS 步骤**之前**增加发布到 GitHub Release。
  - 滚动单一 Release：固定 tag `subtitle-player-latest`、固定资产名 `subtitle_video_player.single.html` → **直链永不变**。
  - 每次构建删旧建新（`gh release delete --cleanup-tag --yes` 再 `gh release create`）。
  - 不保留历史版本（只要最新版）。
  - **去掉了 Artifact**：Artifact 占 Actions 存储配额，而产物有几十兆，会很快烧完配额；对外分发只靠 Release 即可。
- **意义**：Release 又快又稳（GitHub CDN，96MB 约 2 分钟传完），且排在慢吞吞的 DUFS 之前——所以即使 DUFS 上传失败，产物也早已可靠发布。
- 直链：`https://github.com/leolulu/macbookair-files/releases/download/subtitle-player-latest/subtitle_video_player.single.html`

### 2. DUFS / frp 上传约束（已知现状，接受不改）

- DUFS 通过 **frp** 暴露到公网；frp 中转服务器**带宽本身很低（~15 KB/s）**，连接很稳定，但这是物理上限，配置改不动。
- 96MB 的文件经 frp 上传需 ~2 小时，实测常在 **~82 分钟 / 75%** 处被掐断（`curl: (52) Empty reply from server`）。重传无用——每次重头来又是 2 小时。
- **决策**：**不动 DUFS / server 端**。本项目只管项目内的事，不碰服务器。push 到 DUFS 视为「尽力而为」，可靠渠道是 Release（见上）。
- 偶尔也能成功（如 run #33），所以保留现状即可。
- （可选，尚未做）给 DUFS 步骤加 `continue-on-error: true`，让它失败时整个 run 不标红。

### 3. OCR 体积解耦（进行中）

- **问题**：单文件膨胀到 ~96–100MB，**完全是因为构建脚本把 OCR 的模型/库 base64 内联**进了 HTML。
- **体积构成**（原始大小）：

  | 资产 | 大小 | 说明 |
  |---|---|---|
  | 4× ORT `*.wasm` | 40.7M | 运行时只用 1 个变体，却全打进来了 |
  | `ppocr_rec.onnx` + `ppocr_det.onnx` | 15.6M | 识别 / 检测模型 |
  | `opencv.js` | 9.6M | 图像预处理 |
  | `ort.min.js` | 0.5M | ONNX 运行时 |
  | `esearch-ocr.js` + 字典 | ~0.2M | OCR 封装 |
  | **播放器本体** | **~1–2M** | 真正要留下的部分 |

- **关键认知**：**运行时本来就是懒加载**——worker 只在加载 `.sup` 图像字幕、真正触发 OCR 时，才在 `paddle_ocr_worker.js` 的 `ensureReady()` 里去 `importScripts`/`fetch` 这些库和模型。100MB 是构建内联造成的，**不是运行时逻辑问题**。
- **使用背景（为什么这么设计）**：
  1. OCR 只用于**图像字幕（SUP / PGS）**；99% 场景是 SRT / JSON 字幕，根本不碰 OCR。
  2. OCR 开销大：一部电影的 SUP 字幕可能几千条，跑起来 CPU 满载，**户外 / 电池场景基本不可行**（撑不住）。
  3. OCR 有缓存（`js/cache/sup_ocr_cache.js`，机制待最终确认）：同一字幕跑过一次，下次直接读缓存。→ 可以**在家先跑一遍，在外面直接加载缓存**，无需现跑。
- **方案**：把重资产从单文件里拆出去，让 worker 在触发 OCR 时从 **jsDelivr（镜像本仓库文件、自带 CORS）按需下载**（浏览器自动缓存，二次秒开）。基础单文件降到 ~1–2MB（实测 668KB），SRT/JSON 全程不碰网络、不下任何模型。<br>（最初定的是 GitHub Release，后因 Release 无 CORS 改为 jsDelivr，详见下方 2026-06-16 续。）
- **决策记录**：
  - **托管在 jsDelivr（镜像本仓库、自带 `access-control-allow-origin: *`），不放 GitHub Release / DUFS**：GitHub Release 资产**不带 CORS 头**，浏览器里 worker `fetch()` 会被拦（已实测，见下方续）；DUFS 带宽太低。jsDelivr 走 CDN 又快又带 CORS，且直接镜像仓库文件，连发布资源包都省了。
  - **路径写死**，不做成用户可配置——要改是我们改代码，不该让用户操心。
  - **自动注入 commit SHA（不手动维护）**：构建时从 `GITHUB_SHA`（CI）或 `git rev-parse HEAD`（本地）取当前 commit，拼成 `@<SHA>` 基址。配合 workflow 对 `assets/**` 的触发：更新模型 + push → 自动重建并自动钉到新 commit，**没有任何要手动改的 SHA**，漏不了、也不会版本不一致。
  - **wasm 无需手动砍**：ORT 运行时只 fetch 它需要的那 1 个变体，仓库里 4 个 wasm 不会被全下，下载量天然最小。
  - **缓存**：先靠浏览器 HTTP 缓存，不上 Cache API。
  - **彻底替换**：只出轻量版，**不再保留**「全内联、可离线 OCR」的完整版（因为有缓存 + 在家预跑，离线完整版没必要）。
- **改动落点**：① `scripts/build_single-html.js`（停止内联重资产、注入基址、repoint worker 的 `resolveUrl` 与 `ort.env.wasm.wasmPaths`）；② `subtitle_video_player.html` 删掉主线程 eager 加载 `ort.min.js`/`opencv.js` 的死标签（已确认主线程未使用）。
- **红利**：本体降到 1–2MB 后，**第 2 节的 DUFS 上传难题基本自动消失**（每次只传 1–2MB，又快又稳）；那个几十兆的大包只需一次性传到 Release。

### 待确认 / 持续关注

- [x] **CORS（已验证 2026-06-16）**：GitHub Release 资产**无** CORS → 不可用；改用 **jsDelivr**（`access-control-allow-origin: *`）。worker 需要的 8 个资产在 `@<SHA>` 上全部可达且带 CORS。

## 2026-06-16（续）

### OCR 解耦已落地；主机从 GitHub Release 改为 jsDelivr

- **为什么不是 GitHub Release**：实测 Release 资产 CDN（`release-assets.githubusercontent.com`）响应**没有 `access-control-allow-origin`**。浏览器里"点击下载"是顶层下载、不受 CORS 限制（所以平时下得动），但 worker 里 `fetch()` 读字节属于跨域程序化请求，会被浏览器直接拦——与下载速度无关。故 Release 这条路在浏览器里走不通。
- **改用 jsDelivr**：`cdn.jsdelivr.net/gh/<user>/<repo>@<SHA>/...` 直接镜像仓库文件，带 `access-control-allow-origin: *`，无大文件限制（11M 模型正常 206）。资产就是仓库里已有那份，**不用再发布资源包**。
- **实测（本机）**：11M 文件 ~1.1–1.4 MB/s；首次触发 OCR 总下载 ~37MB ≈ **~30 秒一次性**，之后浏览器缓存。对比 DUFS ~15KB/s（光一个 11M 模型就要 ~12 分钟）。
- **运行时去险**：worker 将来要拉的 8 个资产（`ort.min.js`、`opencv.js`、`esearch-ocr.js`、字典、`ppocr_det/rec.onnx`、2 个 wasm）在 `@<SHA>` 上**全部确认可达 + 带 CORS**。⚠️ 真实 OCR 推理（加载 `.sup`）尚未端到端跑过，建议用真实 `.sup` 文件做一次冒烟。
- **改动**：`scripts/build_single-html.js` 注入 `OCR_BASE_URL`（jsDelivr@SHA）、把 worker 的 `resolveUrl` 基址指向其镜像位置（保留 `../../assets/...` 目录结构，`wasmPaths` 随之正确）；删掉 HTML 主线程 eager 加载 `ort.min.js`/`opencv.js` 的死标签（已确认主线程未用 `cv`/`ort`）。
- **结果**：单文件 **~100MB → 668KB**。SRT/JSON/ASS 零网络；OCR 按需。
- **DUFS 红利**：本体只剩 668KB，第 2 节的 DUFS 上传难题自然消失——以后每次只传几百 KB。
- **维护（零手工）**：基址不再写死，构建时自动注入当前 commit SHA（`GITHUB_SHA` / `git rev-parse HEAD`）。更新 `assets/` 下模型/库并 push 即可——workflow 因 `assets/**` 触发，自动重建并自动钉到新 commit。⚠️ 唯一注意：jsDelivr 只能镜像**已 push 到 GitHub** 的 commit，所以本地"已提交但未 push"时若重新构建，OCR 资产会临时 404，push 后即恢复（Action 始终在 push 之后构建，不受影响）。

## 2026-07-14

### AI 对话功能（基于字幕上下文的 AI 问答）

- **功能**：选项区新增「AI对话」按钮，打开可拖动/缩放的浮动聊天窗；字幕全文（带时间轴）作为 system prompt，多轮流式对话。后端依赖 [doubao_backend](https://github.com/leolulu/doubao_backend) 的 `/stream` SSE 接口（POST，`fetch`+`ReadableStream` 手动解析，`EventSource` 不支持 POST）。
- **关键决策**：
  - **浮窗零布局侵入**：全部 `position: fixed` 挂 body，不动 CSS Grid，横/竖 × 正常/反转四种布局变体天然免疫；位置/大小存 localStorage（开合状态**不**持久化，新开/刷新页面默认关闭，要用再点按钮）；视口 ≤700px 自动近全屏。消息区底部有约一半可视高度的滚动缓冲（编辑器式），自动吸底锚定在"最新内容贴底边"，缓冲区留给手动上滚。
  - **每轮重发 system_message**（后端为替换语义，无副作用）：① 后端会话在进程内存里，服务重启后至多丢聊天记忆、字幕背景不丢；② 防剧透开关、超长截断窗口（15 万字符、以当前播放位置为中心滑动）每轮即时生效。
  - **会话 id 前端生成**（`svp-<ts>-<rand>`）+ 每轮 `preserve=true`；后端无清历史接口，「新对话」=换新 id。
  - **防剧透 = 全量字幕 + 提示语约束**（不做硬截断）：模型必须掌握全貌才有能力判断"什么算剧透"；提示语要求不透露当前位置之后的情节、用户明确接受剧透后才可展开。默认开。
  - **杀手级体验——位置注入**：每轮提问自动前缀 `[当前播放到 h:mm:ss，当前字幕:"…"]`，"这句什么意思"类问题免复述（可关）。
  - **时间点闭环**：system prompt 要求 AI 引用字幕时标注 `[h:mm:ss]`，前端正则转成可点击链接直接 seek（经 `getShiftedSubtitleTime` 换算 offset）。
  - **划选字幕气泡**：在字幕区划选文字弹出「查词 / 问AI…」。查词=免打字直发（语境释义模板，含出处句和时间点）；问AI=拉起浮窗+引用条（chip，不污染输入框）。窗口关着也能划选直达，观影流不中断。
  - **字幕更换自动新会话**：不侵入各字幕加载路径，改为发送时对比字幕指纹（条数+首末句），变了自动换会话 id + 提示。
  - **停止语义**与后端对齐：`preserve=true` 只在流正常完成后写历史，客户端 abort 的残缺回答不进上下文；UI 标注"已停止，本轮未计入对话记忆"并回填原问题。
  - **AI 文本渲染安全**：一律 textContent 构建 DOM，不走 innerHTML。
- **顺手修的老 bug**：全局倍速热键 `[` `]` `\` 原先不避让输入框（焦点在输入框打这些字符会变速）；字幕区划选松手会误触 click-to-seek（新增捕获阶段守卫：有选区时拦截字幕区所有点击跳转）。
- **联调工具**：`scripts/mock_ai_backend.py`（纯标准库），模拟 `/stream` SSE 协议，提问包含"触发错误"/"触发500"可测错误分支；`uv run python scripts/mock_ai_backend.py` 默认 11301 端口。
- **已知边界**：video 原生全屏时浮窗不可见（元素全屏遮蔽页面其余部分），不解；页面若以 https 承载而后端为 http 会被 Mixed Content 拦（本地 file:// / http 使用不受影响）。

## 2026-07-16

### AI 回复 Markdown 渲染（已落地）

- **库与集成**：本地托管 `marked@18.0.6` 与 `DOMPurify@3.4.12`，开发版 HTML 直接引用，`scripts/build_single-html.js` 构建时自动内联；没有新增运行时网络依赖。
- **渲染管道**：流式原文 → 临时补齐未闭合代码围栏 → Marked（GFM + `breaks`）→ DOMPurify 白名单消毒 → DOM 后处理时间戳。AI 回复仍按不可信输入处理。
- **安全策略**：禁用 `img`；链接只保留绝对 `http://` / `https://`，相对路径、`mailto:`、`javascript:` 等均移除 `href`；可点击链接统一加 `target="_blank"` 与 `rel="noopener noreferrer"`；脚本、事件属性和危险 URL 会被清除。
- **时间戳兼容**：消毒完成后只遍历普通文本节点，把 `[h:mm:ss]` 转成播放器跳转链接；跳过 `pre`、`code`、`a`，避免代码块和现有链接被误处理。
- **流式策略**：累计全文约每 100ms 重新渲染一次，结束、停止或错误时执行最终渲染；最终渲染继续遵守“用户仍在底部才自动吸底”的规则。消毒移除记录只在最终渲染输出一次，避免流式阶段重复刷屏。
- **降级路径**：Marked 或 DOMPurify 缺失时回退到原纯文本渲染，继续支持时间戳跳转，全程用文本节点构建 DOM。
- **样式**：AI 气泡增加紧凑的标题、段落、列表、行内代码、代码块、引用、表格、分隔线和链接样式；用户气泡继续保持纯文本。
- **联调工具**：mock 回复覆盖标题、强调、列表、表格、引用、代码块、链接协议、时间戳和 XSS 探针；新增“未闭合围栏”测试指令，原“触发错误”“触发500”分支继续保留。
- **验证结果**：开发版与内联单文件均通过浏览器联调；HTTP/HTTPS 白名单、XSS/图片/事件属性清理、代码块时间戳跳过、普通时间戳跳转、停止、HTTP 500、重试、流中错误、未闭合围栏、纯文本降级和最终吸底均已核验。
- **编码修正**：文档头增加 `<meta charset="UTF-8">`，避免无 BOM UTF-8 文件在未声明字符集的 HTTP 服务下被错误解码。

#### 1. 需求背景与约束

原 AI 对话实现为了保证安全，回复气泡始终使用 `textContent`，只额外把 `[h:mm:ss]` 转成可点击时间点。这个方案安全且简单，但真实模型回复通常大量使用标题、列表、强调、表格、引用和代码块；全部按纯文本展示后，信息层级和可读性明显下降。

本轮设计同时受以下约束：

1. **安全边界不能倒退**：AI 回复、字幕和后端错误文本都属于不可信输入；引入 Markdown 后会使用 `innerHTML`，必须先建立可靠的 HTML 消毒边界。
2. **流式输出必须保留**：后端按 SSE `delta` 持续返回不完整文本，任一时刻都可能停在半个强调标记、表格或代码围栏中。
3. **时间点闭环不能丢**：普通正文中的 `[h:mm:ss]` 仍要跳转播放器，同时不能误处理代码块或 Markdown 已生成的链接。
4. **本地优先**：开发版和最终单文件均不能依赖运行时 CDN；无网环境也要保留 AI 聊天本体和 Markdown 渲染能力。
5. **不改变现有接口**：继续沿用现有 SSE、停止、错误重试、消息吸底和单文件构建流程，不要求后端改变返回格式。
6. **功能范围克制**：聊天窗没有展示远程图片的实际需求，因此禁用 `img`；链接可以点击，但只允许绝对 HTTP/HTTPS URL。

#### 2. 技术选型

| 组件 | 固定版本 | 本地文件 | 原始大小 | 职责 |
|---|---:|---|---:|---|
| [Marked](https://github.com/markedjs/marked/blob/v18.0.6/README.md) | `18.0.6` | `assets/vendor/marked/marked.umd.js` | 43,009 B | Markdown / GFM 解析，输出 HTML 字符串 |
| [DOMPurify](https://github.com/cure53/DOMPurify/blob/3.4.12/README.md) | `3.4.12` | `assets/vendor/dompurify/purify.min.js` | 29,209 B | 对 Marked 生成的 HTML 做浏览器 DOM 级 XSS 消毒 |

**选择 Marked 的原因**：

- 浏览器 UMD 产物可以直接用 `<script>` 引入，适合当前无打包器的单页项目，也容易被现有构建脚本内联。
- 支持 GFM，标题、强调、删除线、列表、表格、引用和代码围栏等 AI 常用格式覆盖完整，无需自己维护一套容易出错的正则解析器。
- 官方定位就是轻量、低层、适合浏览器的 Markdown 编译器；当前体积对约 850KB 的最终单文件影响可接受。
- 显式设置 `gfm: true`、`breaks: true`：启用 GFM，并把聊天回复中的单换行按 GitHub 评论式行为渲染成 `<br>`。官方文档注明 `breaks` 依赖 `gfm`。

**Marked 不能单独使用**：其官方 README 明确警告，Marked **不消毒输出 HTML**，并直接推荐对 `marked.parse()` 的结果使用 DOMPurify。AI 可以输出原始 HTML，因此把 Marked 结果直接交给 `innerHTML` 会重新打开 XSS 风险。

**选择 DOMPurify 的原因**：

- 它使用浏览器自身的 HTML 解析和 DOM 能力处理畸形标签、命名空间和事件属性等场景，比手写标签过滤或正则替换可靠。
- 支持显式 `ALLOWED_TAGS`、`ALLOWED_ATTR`、URL 正则和 hooks，能够把聊天窗所需能力收缩到最小集合。
- 提供 `isSupported`，在不支持安全消毒的环境中可以明确走纯文本降级，而不是冒险渲染 HTML。
- `DOMPurify.removed` 只作为诊断记录；遵循官方说明，不把它用于任何安全判断。

**没有采用的路径**：

- **手写 Markdown 正则**：嵌套强调、列表、表格、转义和代码围栏很快会变成不完整的自制解析器，维护成本和边界错误都高。
- **只用 Marked**：缺少 HTML 消毒，不满足安全约束。
- **CDN 运行时加载**：会破坏本地优先和单文件离线能力；当前两个 vendor 文件合计只有 72,218 B，没有必要引入网络依赖。
- **每个 delta 直接追加 HTML / DOM**：后续字符可能改变前面 Markdown 的语义，例如三个反引号会把已有普通段落重新解释成代码块；增量拼接需要维护复杂解析状态，也更难保持统一的消毒边界。
- **每个 delta 立即全量解析**：结构正确，但 token 频率可能远高于屏幕刷新需要，会造成重复解析、消毒和 DOM 抖动；最终采用约 100ms 节流。

#### 3. 完整渲染管道与安全不变量

每次渲染严格按以下顺序执行：

```text
SSE delta 原文累计
  → 临时修复未闭合代码围栏
  → marked.parse（GFM + breaks）
  → DOMPurify.sanitize（标签 / 属性 / URL 白名单）
  → 将“已消毒 HTML”写入 AI 气泡 innerHTML
  → 遍历安全 DOM，把普通文本中的时间戳替换为受信任的跳转节点
```

核心安全不变量：**只有 `DOMPurify.sanitize()` 的返回值可以进入 `innerHTML`**。用户问题、后端错误、停止提示和重试按钮仍使用 `textContent` / `createElement` 构建。

Marked 解析在消毒之前，原因是 Markdown 解析本身会生成新的 HTML；如果先消毒原始 Markdown、再交给 Marked，解析器仍可能在安全步骤之后生成危险标签。当前顺序与 Marked 官方给出的 `DOMPurify.sanitize(marked.parse(...))` 用法一致。

DOMPurify 官方提醒，消毒后继续让其他库改写 HTML 可能破坏安全性。当前消毒后的后处理只使用受信任代码创建文本节点和时间戳 `<span>`，不会重新解析 AI 字符串、复制 AI 属性或扩大白名单；已有 `pre`、`code`、`a` 子树直接跳过。

#### 4. DOMPurify 白名单与链接策略

允许的标签只覆盖聊天排版需要：

- 文本：`p`、`br`、`strong`、`em`、`b`、`i`、`del`、`s`、`span`
- 结构：`ul`、`ol`、`li`、`blockquote`、`h1`–`h6`、`hr`
- 代码：`code`、`pre`
- 表格：`table`、`thead`、`tbody`、`tr`、`th`、`td`
- 链接：`a`

允许的属性只有 `href`、`title`、`class`、`start`、`align`：

- `class` 保留代码围栏的 `language-*` 等语义类名；当前尚未引入语法高亮器。
- `start` 保留有起始序号的有序列表。
- `align` 保留 GFM 表格对齐信息。
- `data-*` 通过 `ALLOW_DATA_ATTR: false` 全部禁用。
- `style`、`id`、事件属性和任意自定义属性均未放行。

`img`、`svg`、`math`、`iframe`、`video`、`audio`、`style`、`script`、表单控件等均不在白名单。GFM task list 可能生成 `<input type="checkbox">`，该控件会被清除，只保留列表文字；这是刻意的安全和交互取舍。

链接使用 `ALLOWED_URI_REGEXP: /^https?:\/\//i`，只保留绝对 HTTP/HTTPS：

- `https://example.com`、`http://example.com`：保留并可点击。
- 相对路径、协议相对 URL、`mailto:`、`tel:`、`data:`、`file:`、`javascript:`：移除 `href`，可见文字仍保留。
- `afterSanitizeAttributes` hook 只对消毒后仍有 `href` 的 `<a>` 添加 `target="_blank"` 和 `rel="noopener noreferrer"`。

#### 5. 流式 Markdown 策略

SSE 每个 `delta` 只追加到 `accumulated` 原文，不直接改 HTML。最多约每 100ms 执行一次“累计全文全量重解析”：

- **为什么全量重解析**：Markdown 是上下文相关格式，后续字符可以改变前面内容的结构；每次从完整原文重建 DOM，结果始终等价于对当前回复做一次正常 Markdown 渲染。
- **为什么节流到约 100ms**：把解析、消毒和 DOM 替换控制在最高约 10 次/秒，保留可感知的流式效果，同时避免每个细碎 token 都触发重排。
- **吸底语义**：渲染前先记录用户是否仍贴近最新内容；只有原本在底部才调用 `scrollToBottom()`，用户主动上滚阅读时不会被拉回。
- **最终渲染**：正常完成、停止和错误都先取消挂起的 timer，再用完整累计文本做一次最终渲染，确保最后一个尚未进入节流周期的 chunk 不丢失。成功和错误路径在最终 DOM 变化后再次恢复吸底位置。
- **消毒日志**：流式中间帧不打印 `DOMPurify.removed`；最终渲染最多记录一次，避免同一恶意片段在每次重解析时刷几十条警告。

##### 未闭合代码围栏

模型流式输出代码块时，开始围栏通常先于结束围栏到达。若直接解析，后续内容可能暂时全部被吞进代码块。`repairUnclosedFences()` 检测行首允许 0–3 个空格的反引号型或波浪号型围栏；围栏总数为奇数时，仅在**本次送给 Marked 的副本末尾**补与最后一个围栏同类型的结束标记。原始累计文本不修改，真正的结束围栏到达后自然恢复正常解析。

#### 6. 时间戳后处理

原功能会把 `[h:mm:ss]` 转成可点击播放器跳转。本轮没有把时间戳写成 Marked 扩展，而是放在“解析 + 消毒”之后做 DOM 后处理：

1. 使用 `TreeWalker` 查找文本节点，先收集、后替换，避免遍历过程中修改树导致漏节点。
2. 只处理匹配 `[m:ss]` / `[h:mm:ss]` 的文本。
3. 跳过 `pre`、`code`、`a` 内部，防止代码示例和已有链接被嵌套改写。
4. 通过 `createTextNode`、`createElement` 和 `DocumentFragment` 构建节点，不再次使用 `innerHTML`。
5. 点击后继续调用 `getShiftedSubtitleTime()` 应用字幕 offset；“点击字幕播放”开启时同时恢复视频播放。

这种分层让 Markdown 解析器只负责 Markdown，播放器专属语义仍由现有代码负责，后续更换解析库时也不需要重写 seek 逻辑。

#### 7. 降级、停止与错误路径

- 页面初始化时同时检查 `marked`、`DOMPurify` 和 `DOMPurify.isSupported`。任一条件不满足，AI 回复回退到原纯文本 + 时间戳渲染，全程不进入 `innerHTML`。
- `marked.parse()` 若异常，同样对当前回复降级为纯文本，不让单条畸形回答破坏整个聊天状态。
- 用户停止时：保留已收到内容并做最终渲染，追加“已停止，本轮未计入对话记忆”，再把原问题回填输入框。
- 请求失败或流中 error 时：先安全渲染已累计 Markdown，再用独立 DOM 节点追加错误文字和“重试”按钮；错误文本不会拼进 Markdown 后重新解析。
- 重试复用原请求参数和原气泡，不额外创建一条用户消息。

#### 8. 本地 vendor 与单文件构建

开发版脚本顺序固定为：jQuery → jQuery UI → Marked → DOMPurify → 项目脚本。AI 初始化时两个全局对象已经可用。

`scripts/build_single-html.js` 将两个 vendor 文件加入 `inlineScripts`，随后用 `replaceOnce()` 精确替换对应 `<script src>`：

- 开发版保留独立文件，便于调试和版本识别。
- 最终 `subtitle_video_player.single.html` 内联完整库，不产生运行时 Markdown CDN 请求。
- 目标标签缺失时构建会直接失败，避免悄悄产出依赖未内联的“伪单文件”。`replaceOnce()` 只替换第一个匹配项，因此 HTML 中必须保持每个 vendor 标签唯一；维护时还要检查最终产物没有残留对应的外部 `<script src>`。
- 2026-07-16 实测最终单文件约 851KB；构建验证在内存截获产物完成，没有留下测试构建文件。

HTML 头部同时增加 `<meta charset="UTF-8">`。源文件为无 BOM UTF-8；显式声明后，即使普通 HTTP 服务未在 `Content-Type` 中带 charset，浏览器也能按 UTF-8 解码中文。

#### 9. 样式策略

Markdown 样式全部限定在 `.ai-chat-msg-ai.ai-chat-md` 下，避免影响播放器其他区域。AI 气泡使用正常空白折叠；用户气泡继续保留 `pre-wrap`，确保用户输入原样显示。

- 标题缩小到聊天尺度，避免 `h1` 占据过多空间。
- 段落、列表和列表项使用紧凑间距。
- 行内代码与代码块使用等宽字体；代码块横向滚动，不撑宽浮窗。
- 表格使用块级横向滚动，窄窗口下不破坏整体布局。
- 引用使用左边框和浅色背景；链接沿用时间点链接的蓝色视觉语言。
- 当前只保留代码语言 class，不引入 syntax highlighter；以后若增加高亮，需要单独评估体积、样式和消毒白名单。

#### 10. 联调与安全回归矩阵

| 场景 | 期望 / 实测结果 |

- 包含“触发500”：直接返回 HTTP 500。

#### 11. 维护与升级清单

以后升级 Marked 或 DOMPurify 时，应完成以下步骤：

1. 从明确版本 tag 获取官方 UMD / minified 文件，保留文件内 license 和版本头；不要直接跟随 `latest`。
2. 保持当前 vendor 路径和 script 顺序，或同步修改 HTML 与构建脚本中的所有 `replaceOnce()` 锚点。
3. 重新核对 Marked 的 GFM / `breaks` 行为，以及 DOMPurify 的默认行为和 hook API；安全库升级不能只看“页面还能显示”。
4. 保持显式标签、属性和 URL 白名单；新增 Markdown 能力时先确认是否需要扩大白名单，默认不放行新标签。
5. 运行 mock 的 Markdown、XSS、URL、时间戳、停止、错误、降级和未闭合围栏用例。
6. 同时验证开发版和真正的内联单文件；只测开发版不足以证明构建脚本已正确包含依赖。
7. 检查浏览器 console：消毒探针应有且仅有一次最终 warning，不应出现脚本加载或运行错误。

当前刻意保留的边界：不渲染图片、不提供可交互 task-list checkbox、不做代码语法高亮、只允许绝对 HTTP/HTTPS 链接。若未来回答特别长、100ms 全量重解析出现可感知性能问题，再基于实际 profiling 考虑更复杂的增量 token 渲染；在此之前优先保持当前实现的正确性和安全边界。

### AI 悬浮对话框：四边四角八方向缩放

#### 1. 背景与方案选择

原实现直接在 `.ai-chat-window` 上使用浏览器原生 `resize: both`。在当前桌面浏览器中，它只在右下角提供一个原生缩放入口，命中范围、图标样式和悬浮反馈也由浏览器控制，无法形成类似 Windows 窗口的四边四角体验。

本轮关闭原生缩放，改为 8 个显式手柄：上、右、下、左以及右上、右下、左下、左上。手柄只负责尺寸调整；窗口位置拖动仍沿用原有 `#aiChatHeader` 指针逻辑，标题栏按钮和正文区域不会触发移动。

没有采用把整个窗口监听成一个大拖拽区域的方案，原因是边缘缩放与标题栏移动需要明确分层：边缘优先命中尺寸调整，标题栏内部继续移动窗口，正文选择、输入和按钮交互维持原行为。

#### 2. 命中区域与视觉提示

- 四条边使用 8px 宽的透明命中带，四个角使用 14×14px 命中区；命中区比可见提示更大，鼠标无需精确压在 1px 边框上。
- 边缘采用标准 `ns-resize` / `ew-resize` 光标，角落采用 `nesw-resize` / `nwse-resize`，方向语义与桌面窗口一致。
- 静止时不常驻额外边框，保持悬浮窗原有的轻量外观。鼠标进入边缘后，在对应边的中部显示一段 42×2px 蓝色短线；进入角落后显示 7×7px 蓝色 L 形角标。
- 提示色沿用播放器现有的蓝色交互语言（`#4d8dff`），并使用短时透明度过渡和轻微光晕，帮助用户发现命中位置，又不会形成持续的视觉噪声。
- 调整进行中，当前手柄提示提高到完全不透明，窗口阴影略微增强并出现一层很淡的蓝色描边；页面临时禁用文本选择，光标在指针越过窗体边界后仍保持当前调整方向。

#### 3. 几何计算与边界约束

指针按下时记录窗口的 `left`、`top`、`width`、`height` 和按下坐标。移动时按方向只更新对应分量：

| 方向 | 变化量 | 固定不动的对边 |
|---|---|---|
| 东 / 南 | `width += dx` / `height += dy` | 西 / 北 |
| 西 / 北 | `left += dx, width -= dx` / `top += dy, height -= dy` | 东 / 南 |
| 四个角 | 组合对应的横向与纵向规则 | 两条相对边 |

北侧和西侧缩放的关键是先计算新尺寸，再用原始 `right` / `bottom` 反推新位置，从而保证用户视觉上抓住的边跟随指针、对边保持稳定。开始调整时把当前几何转换为显式的 `left/top/width/height`，并清除 `right/bottom`，避免 CSS 定位约束互相竞争。

尺寸继续遵守 `.ai-chat-window` 的现有约束：最小 300×260px，最大宽高取 CSS `max-width` / `max-height` 与当前视口可用范围中的较小值。所有结果通过 clamp 限制，防止快速拖动越过最小尺寸或把窗口扩展到视口之外。窗口使用 `box-sizing: border-box`，保存和恢复的 DOM 外框尺寸与写入的 CSS 尺寸保持一致。

#### 4. 指针生命周期与状态持久化

- 每个手柄只监听自己的 `pointerdown`；一次调整开始后，`pointermove`、`pointerup`、`pointercancel` 临时挂在 `window` 上，因此指针离开 8px 命中带或窗体范围后仍能连续调整。
- `setPointerCapture()` 作为可用时的增强，调用放在保护性 `try/catch` 中；监听注册不依赖捕获成功，避免个别浏览器或自动化环境拒绝捕获后留下“已按下但不再移动”的状态。
- 结束或取消时统一移除临时监听、活动样式和页面光标，再调用现有 `captureGeometry()` / `persistWindowState()`。现有 `ResizeObserver` 仍作为延迟持久化兜底。
- 沿用 `AI_CHAT_WINDOW_STATE_V1`，没有增加新的存储键或迁移格式；用户调整后的尺寸和位置在收起、重新打开及页面重载后继续恢复。

#### 5. 响应式策略

`max-width: 700px` 的小屏规则保持原设计：悬浮窗声明为 96vw×96vh，位置贴近视口边缘；现有 `max-height: 92vh` 会把实际高度限制为 92vh，标题栏不参与拖动。本轮同时把 8 个缩放手柄全部隐藏；触屏窄屏继续使用固定的大面板布局，避免透明边缘与页面手势、输入滚动发生冲突。

#### 6. 回归验证

在 1280×720 桌面视口中逐一实拖 8 个方向：

- 东、南只改变宽或高；西、北改变尺寸的同时保持右边或下边固定。
- 东南、西南、东北、西北均正确组合横纵两轴，对角方向结束后活动状态能完整清理。
- 宽度向内拖动会停在 300px，高度向内拖动会停在 260px。
- 页面重载后，窗口的 `left/top/width/height` 与调整结束时一致。
- 680×800 窄屏视口命中 `max-width: 700px` 规则，8 个手柄计算样式为 `display: none`，窗口保持约 96vw×92vh 的移动端布局。
- 浏览器控制台没有新增运行时错误；内联 JavaScript 语法检查和单文件构建均通过。

### AI 服务商与模型动态发现、级联选择及会话锁定

#### 1. 后端能力与前端接入边界

`doubao_backend` 新增 `GET /models`，返回当前进程已加载配置中可以手动选择的服务商和模型：

```json
{
  "providers": [
    { "id": "zhipu", "models": ["glm-4.7", "glm-4-flash"] },
    { "id": "doubao", "models": ["ep-a", "ep-b"] }
  ]
}
```

该接口属于**本地配置发现**：它不会连接上游服务商枚举全部模型，也不会验证 API Key 或模型当前是否在线。前端直接使用返回的 `id` 作为请求 `provider`，使用 `models` 元素作为请求 `model`，不再维护豆包、智谱、DeepSeek 等硬编码名单。这样后端增加服务商、Chat Completion 别名或模型后，只需更新配置并重启后端，播放器即可自动呈现。

#### 2. 三种路由模式

Provider 和 Model 均保留值为空字符串的“默认”选项，请求体严格按以下状态构造：

| Provider | Model | 请求字段 | 后端行为 |
|---|---|---|---|
| 默认 | 默认（禁用） | 不发送 `provider` / `model` | 使用后端完整默认服务商与模型回退链 |
| 指定服务商 | 默认 | 只发送 `provider` | 使用该服务商配置的模型回退链 |
| 指定服务商 | 指定模型 | 同时发送 `provider` / `model` | 固定服务商与模型，保留后端 API Key 切换和重试 |

Provider 为默认时 Model 必须禁用，避免界面形成“默认服务商 + 指定模型”这种后端明确拒绝的组合。切换 Provider 时 Model 自动恢复默认，再按新 Provider 的 `models` 重建选项；具体模型名称原样展示并提交，保留大小写。

Provider 与 Model 在设置面板中采用同一行的左右两列布局：整组左右各保留 `6px` 内边距，两列之间保留 `14px` 间隔，每列内部继续使用“标签 + 下拉框”的水平结构。两列都使用 `minmax(0, 1fr)`，下拉框同时设置 `min-width: 0`，使两侧等分剩余空间并允许控件在悬浮窗收窄时正确压缩；该区域不随窄窗口改回垂直排列，避免重新占满两整行。模型发现或会话锁定提示则与整组控件的左右边界对齐。两个下拉框统一使用 `28px` 高度，在扩大点击区域的同时避免相对其他设置项过于厚重；标签文字包裹在同高的 `inline-flex` 容器中并垂直居中，避免原生 Select 的行盒与裸文本节点产生视觉错位。

#### 3. 模型发现状态机

页面初始化后会对当前后端 URL 发起一次 `GET /models`；设置面板在上次加载失败时再次打开会自动重试，后端 URL 变化也会清空旧选项并强制重新发现。

前端维护 `idle / loading / ready / error` 四种状态：

- `loading`：Provider 和 Model 只显示默认值并禁用，提示“正在从后端获取可用服务商和模型”。
- `ready`：按接口顺序创建 Provider；选中 Provider 后创建该项对应的 Model。
- `ready` 但列表为空：保持自动模式，提示没有发现可手动选择的服务商。
- `error`：清除可能已经失效的 Provider/Model 设置，回到自动模式；设置面板显示失败或超时提示，再次打开时重试。默认流式聊天仍可继续尝试调用后端，兼容暂时缺少 `/models` 的旧服务实例。

发现请求使用独立 `AbortController` 和 6 秒超时。后端 URL 连续变化时会中止上一次请求；递增序号保证旧请求即使晚到，也不能覆盖新 URL 的结果。响应进入界面前还会校验 `providers` 数组、Provider ID、Model 数组，去掉空值和重复值；选项全部通过 `createElement` / `textContent` 创建，不解析后端返回的 HTML。

#### 4. 设置持久化与旧版本兼容

继续沿用 `AI_CHAT_SETTINGS_V1`，只在原对象中增加 `model` 字段，不升级存储键：

- 旧版本没有 `model` 时自然按空字符串处理，即 Model 默认。
- 保存的 Provider/Model 必须在新一次发现结果中仍然存在，才会恢复到下拉框。
- 后端删除或重命名配置后，失效值会自动回到默认并立即重写本地设置，避免界面显示默认、请求却暗中携带旧模型。
- 后端 URL 变化时 Provider 和 Model 同时恢复默认，防止把上一台后端的配置名称带到新后端。

#### 5. 会话开始后的锁定策略

后端只在创建会话时读取 Provider/Model，同一会话 ID 后续传入新值不会切换客户端。为了避免界面显示已经切换、实际仍使用旧模型，播放器采用显式锁定：

- 第一条通过前置检查、即将真正发送的消息会把 `conversationStarted` 置为 `true`。
- 对话开始后，后端 URL、Provider 和 Model 全部禁用，并提示“点击新对话后可重新选择”。后端 URL 同样锁定，因为切到另一台后端会产生与中途切模型相同的上下文错位。
- 请求失败、停止或重试不会解锁；用户已经开始了该会话，若要改路由必须明确点击“新对话”。
- 点击“新对话”会生成新会话 ID、清空消息并解锁设置。
- 字幕更换触发原有的自动新会话时也会先解锁；随后若用户立即发送第一条消息，再按正常流程锁定。

设置变化本身不会自动清空聊天，也不会暗中创建新会话，从而避免用户只是查看或误触下拉框就丢失当前对话。

#### 6. mock 与回归入口

`scripts/mock_ai_backend.py` 新增：

- `GET /models`：返回两个 Provider、每个两个 Model，带与真实后端一致的 CORS 和 JSON Content-Type。
- `POST /models`：返回 405，用于确认接口只读语义。
- mock 回答显示实际收到的 `provider` / `model`，便于联调三种请求模式。

浏览器回归覆盖：初始双默认、Provider 动态加载、Provider→Model 级联、Provider 级“默认”模型、具体模型选择、页面重载后的选择恢复、Provider 恢复默认后 Model 禁用、后端离线时失败回退、重新打开设置自动恢复，以及控制台无新增运行时错误。

## 2026-07-17

### 查词 Prompt 的作品语境优先与单义项约束

#### 1. 调整范围

本次只调整划词气泡“查词”入口生成的用户消息，不改变普通 AI 问答共用的系统 Prompt。查词请求仍会获得系统消息中的视频信息、字幕全文、当前播放位置、防剧透规则和时间点引用能力；新增内容用于告诉模型如何排列这些信息与自身知识的优先级。

#### 2. 三层知识优先级

查词回答按以下顺序组织依据：

1. 当前字幕、本句场景和本片上下文，包括说话人、交流对象、人物关系、剧情作用、片中其他同义项用例，以及俚语、梗、双关和文化指涉。
2. 模型可靠掌握的该作品设定、世界观、人物关系和背景知识。
3. 选中词语“当前义项”的通用语言知识。

这样可以先回答“它在这个作品、这个场景中是什么意思”，再补充语言学习信息。人物称号、地点、组织、能力、物品及虚构概念会优先按作品内身份和作用解释。

#### 3. 影视语言的文化层

俚语、梗、双关、反讽、委婉表达、文化典故、地域用法和时代色彩都归入作品语境解释，优先级高于通用词义与搭配。影视对白往往借助这些元素表达人物身份、关系、情绪和潜台词，因此 Prompt 要求在存在相关迹象时着重说明其含义、文化背景、语气、潜台词及当前场景中的表达效果。

这项要求带有条件：只有确实存在或有充分上下文依据时才展开，没有明确依据时不能为了套用模板而强行解释俚语、梗或双关。通用语言补充阶段不再承担这部分内容。

#### 4. 字幕之外的作品知识

允许模型调用其自身已知的作品设定，但增加三项约束：

- 补充内容必须与当前字幕一致；字幕与模型记忆出现冲突时，以当前提供的字幕为准。
- 仍需遵守防剧透设置，不能因调用作品知识而提前透露当前播放位置之后的信息。
- 无法可靠识别作品或确认设定时必须说明不确定，禁止补造世界观、人物关系或剧情事实。

#### 5. 只解释当前义项

通用语言说明放在作品语境解释之后，并严格限定为本句实际使用的一个义项。可以说明该义项的通用含义、词性、语气、常见搭配和固定词组；片中其他时间点、搭配或例句也必须属于同一义项。Prompt 明确禁止罗列其他词典义项，以及提供属于其他义项的搭配或例句。

当作品内用法与通常用法存在差异时，回答先给出作品内含义，再简要指出差异。最后要求按相关性组织内容，无需机械覆盖每个项目，避免简单词语也产生冗长的模板化回答。
